# 模型如何提取与精准归纳 Skills

## 一、Skill 的定位

Skill 不是一次性提示词，而是一类可复用工作流程的标准化说明。它用于告诉模型：

- 什么时候使用这项能力；
- 任务应该按什么步骤完成；
- 输出结果需要满足哪些标准；
- 哪些表达、格式或动作需要避免；
- 交付前应该如何自查。

可以把 Skill 理解为“面向 AI 的岗位操作手册”。

## 二、Skill 与普通提示词的区别

| 对比项 | 普通提示词 | Skill |
|---|---|---|
| 使用频率 | 一次性 | 长期复用 |
| 适用范围 | 当前任务 | 一类任务 |
| 结构 | 临时说明 | 固定流程与标准 |
| 是否可沉淀 | 容易丢失 | 可版本管理 |
| 是否可回退 | 难回退 | 可通过 Git 回退 |
| 适合内容 | 临时要求 | 工作规范、审查标准、输出边界 |

## 三、从对话中提取 Skill 的五步方法

### 第一步：区分长期规则和临时要求

长期规则适合写入 Skill，例如：

- 客户文档不能出现内部分析。
- PPT 图片页默认不加页码。
- 标题位置和字号需要统一。
- 图片生成后必须检查错别字。
- 标书资质不确定时留空。

临时要求不适合写入 Skill，例如：

- 某一页标题改成某句话。
- 某个客户本次要求增加某个模块。
- 某个项目的金额、周期、人员、编号。
- 某张图片中的局部修改。

判断标准：如果这条规则未来还会反复使用，就可以沉淀；如果只服务于当前任务，就不要沉淀。

### 第二步：归类到具体 Skill

不要把所有规则放到一个大 Skill 中，应根据任务类型归类。

| 规则 | 应归入 Skill |
|---|---|
| 客户文档不能有内部讲解口吻 | customer-ready-document |
| PPT 不加页码和 Logo | ppt-visual-page |
| 报价表不写人天单价 | bid-response |
| 官网不能照搬参考网站 | website-saas-design |
| 拜访记录要有下一步动作 | customer-visit-and-channel |

### 第三步：提炼成可执行规则

差的规则写法：

> 文档要高级一点。

好的规则写法：

> 客户正式文档应使用正式、稳重、克制的表达，不使用口语化、内部分析、教学式说明和过度承诺。

差的规则写法：

> PPT 要好看。

好的规则写法：

> 同一组 PPT 图片页中，标题位置、高度、字号必须统一；默认不加页码和 Logo；总结栏宽高应保持一致。

### 第四步：增加正例和反例

Skill 里只写抽象规则，模型有时理解不稳定。增加正例和反例可以提升稳定性。

错误表达：

> 你可以这样给客户讲，这个平台主要是把后端能力沉淀下来。

正确表达：

> 平台围绕统一底座能力建设，形成业务流程、权限体系、数据服务与系统集成的统一支撑基础。

### 第五步：增加输出前审查清单

每个 Skill 都应该有审查清单。

客户正式文档示例：

- 是否可以直接发给客户？
- 是否存在内部分析口吻？
- 是否存在过度承诺？
- 是否适合客户管理层阅读？

PPT 图片页示例：

- 标题位置是否统一？
- 是否有错别字？
- 是否默认没有页码和 Logo？
- 插图是否过于抢眼？

## 四、精准归纳 Skill 的原则

### 1. 一个 Skill 只解决一类问题

推荐拆分：

- customer-ready-document
- ppt-visual-page
- bid-response
- website-saas-design
- visual-poster

### 2. description 要写清楚触发场景

`description` 是模型判断是否使用 Skill 的关键。它应该写清楚任务类型、适用对象和输出边界。

### 3. 规则要具体，不要只写感受

不要只写“高级”“好看”“专业”“大气”“不要 AI 感”。

应拆成具体规则：

- 标题字号统一。
- 总结栏宽高统一。
- 默认浅色背景。
- 不加页码。
- 不出现内部口吻。
- 不做过度承诺。

### 4. 保留边界

Skill 必须写清楚不适用场景。

例如：

- PPT 图片页 Skill 不用于 Word 文档。
- 客户正式文档 Skill 不用于海报视觉设计。
- 一次性项目要求不写入长期 Skill。

### 5. 支持版本记录

每次更新 Skill 时，应同步更新 `CHANGELOG.md`。建议记录新增、调整、删除和回退原因。

## 五、对话结束后的沉淀判断

每次项目结束后，可以从以下角度判断是否更新 Skill：

1. 本次是否出现长期可复用规则？
2. 规则属于哪个 Skill？
3. 是否已有类似规则？
4. 本次属于新增、调整还是删除？
5. 是否和已有规则冲突？
6. 是否需要加入正例或反例？
7. 是否需要更新审查清单？

## 六、回退与修正

如果更新后的 Skill 不符合原来的工作流程，不要直接删除整个 Skill。应先判断问题：

| 问题 | 处理方式 |
|---|---|
| 该用时没用 | 优化 description |
| 不该用时乱用 | 收窄适用场景 |
| 风格不像 | 增加正例和反例 |
| 规则冲突 | 增加优先级或拆分 Skill |
| 更新效果差 | 回退到旧版本 |

## 七、最终原则

Skill 的目标不是把所有想法都存进去，而是沉淀稳定、可复用、能提升交付质量的工作标准。

核心原则：

> 只沉淀长期规则，不沉淀临时要求；只写可执行标准，不写模糊感受。
